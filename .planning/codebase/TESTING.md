## Manual Testing Patterns

The codebase relies on manual, integration-style testing rather than unit tests. Key patterns:

**1. CLI Command Testing:**
- Commands invoke actual external services (Firecrawl, Google Search Console, etc.)
- Dry-run modes provided to test without execution: `--dry-run` flag in clone command
- Integration tested end-to-end by running command with real or test data

**Example from `packages/cli/src/commands/clone.ts`:**
```typescript
static override flags = {
  page: Flags.string({ description: 'Clone a single page by URL or path' }),
  'dry-run': Flags.boolean({ description: 'Print the agent prompts without running Claude Code' }),
  'no-verify': Flags.boolean({ description: 'Skip the screenshot verification loop' }),
};
```

**2. File I/O and Data Transformation Testing:**
- Data loading and transformation functions tested indirectly through CLI commands
- Example: `readClientConfig()`, `loadPages()`, `loadAuditPasses()` called by dashboard and CLI
- Validation occurs at runtime via error handling (thrown errors if data malformed)

**3. Verification Loops:**
- Screenshot verification loop in `packages/cli/src/clone/verify.ts` — iteratively compares cloned pages to source
- Uses Playwright headless browser for visual comparison
- CLI prompts user/Claude for corrections until convergence or iteration limit

**Example from `packages/cli/src/clone/verify.ts`:**
```typescript
export async function verifyClonePage(opts: VerifyOptions): Promise<VerifyResult> {
  // Lazy-import Playwright so users without it can still use --no-verify
  let chromium: typeof import('playwright').chromium;
  try {
    const pw = await import('playwright');
    chromium = pw.chromium;
  } catch {
    opts.log('  [verify] Playwright not installed. Skipping verification loop.');
    return { iterationsRun: 0, status: 'no-chromium', attempts: [] };
  }
  // ... starts dev server, polls URL, takes screenshots, calls Claude Code agent
}
```

## Mocking and Test Doubles

**No formal mocking library present** — Instead:

**Environment Variables for Configuration:**
- Services use environment variables to control behavior
- Example: `CLAUDE_BIN`, `FIRECRAWL_API_KEY`, `GOOGLE_SERVICE_ACCOUNT_KEY`
- Tests/dry-runs can override via environment or use `--dry-run` flags

**Lazy Imports for Optional Dependencies:**
- Playwright imported dynamically so it's not required if verification is skipped
- Non-fatal error handling for missing dependencies: `try { import('playwright') } catch { ... }`

**Fire-and-Forget Non-Fatal Operations:**
- Logging/usage events wrapped in try-catch that silently fails if service unavailable
- Allows tests to run without Supabase or other optional backends

**Example from `packages/core/src/usage/logger.ts`:**
```typescript
export async function logUsageEvent(event: UsageEvent): Promise<void> {
  const cost = calcCost(event);
  const record = { ...event, cost_usd: cost, created_at: new Date().toISOString() };

  // Supabase insert — skipped if env var is not set
  const supabaseUrl = process.env['UPRIVER_SUPABASE_URL'];
  const supabaseKey = process.env['UPRIVER_SUPABASE_SERVICE_KEY'];

  if (supabaseUrl && supabaseKey) {
    try {
      await fetch(`${supabaseUrl}/rest/v1/usage_events`, { ... });
    } catch {
      // non-fatal
    }
  }
}
```

## Test Types

**Integration Testing:**
- Full CLI command execution with real or stubbed external services
- Example: `upriver scrape` runs Firecrawl scraping end-to-end
- Flags like `--dry-run` allow testing CLI argument parsing without executing
- Verification loop tests cloning end-to-end with visual comparison

**No Unit Tests:**
- Functions tested through integration (CLI commands using them)
- Pure utility functions verified by type checker (TypeScript strict mode)
- Error handling verified through error paths in commands

**No E2E Tests:**
- Not applicable — CLI tools don't have web UI to test
- Integration with external services (Firecrawl, Google APIs) tested in scrape/audit commands

## Error Testing

**Pattern:**
- Functions throw descriptive errors on failure
- Errors propagate up to CLI which handles with `this.error()`
- No try-catch blocks except for non-fatal operations

**Example from `packages/core/src/config/client-config.ts`:**
```typescript
export function readClientConfig(slug: string, base = './clients'): ClientConfig {
  const path = configPath(slug, base);
  if (!existsSync(path)) {
    throw new Error(`No client config found at ${path}. Run: upriver init <url> --slug ${slug}`);
  }
  return parseYaml(readFileSync(path, 'utf8')) as ClientConfig;
}
```

**Error Recovery Examples:**
- Silent failures for non-critical operations: `try { await logUsageEvent(...) } catch { }`
- Graceful degradation: `Playwright not installed? Skip verification loop.`
- User-facing error messages from CLI: `this.error('No page matched ...')`

## Test Patterns Summary

Since automated testing is not present, the codebase relies on:

1. **Type Safety** — TypeScript strict mode catches errors at compile time
2. **Linting** — ESLint enforces code quality and consistency
3. **Integration Testing** — CLI commands tested end-to-end with real data/services
4. **Manual Verification** — Humans inspect outputs (screenshots, generated files, etc.)
5. **Error Handling** — Explicit error throws with descriptive messages guide troubleshooting
6. **Dry-Run Modes** — Commands provide `--dry-run` to test without side effects
7. **Optional Dependencies** — Non-essential features (Playwright) gracefully degrade if unavailable

**Future Improvements:**
- Add Jest or Vitest for unit testing core utilities
- Add test fixtures for audit passes and data transformation functions
- Add E2E tests for critical CLI commands with stubbed external APIs
