# Build Spec 09: `upriver generate --provisioning` (I01–I09 provisioning artifacts, M5)

Component: M5 — provisioning generation of the client-facing **I01–I09** artifacts (Claude Project, Skills, MCP config, Cowork routines, Claude in Chrome, Claude Code, Account/Governance, Styles/Memory/Prefs, Artifact templates) from the profile + the already-generated docs. Extends the M2 DAG-batch engine (Build Spec 08) onto the i-series scope; it does **not** rewrite the single-doc or batch core. The I-series has a "different output shape than M2's prose docs" (PRD §8 M5): each artifact is a generatable config/reference PLUS a per-artifact operator runbook PLUS explicit cannot-generate (click-ops) markers, because most provisioning steps happen inside the client's Anthropic account.
Built: tier-3, fresh Claude Code session, after M2 (`generate --all`) and M4 (`profile verify`) merged to `main`. The i-series is the most HV-dense scope in the system (PRD §6), so M4's verify flow is a hard prerequisite.
Source of truth: `01-prd.md` §3.4 (the I01–I09 coverage map: which profile sections + which docs each consumes), §6 (provisioning hooks: DAG placement, output-format-varies-by-API-surface, HV densest here), §8 M5 (depends on M2 + M4); `specs/02-generate-spec.md` (the single-doc engine), `specs/08-generate-all-spec.md` (the DAG-batch engine this extends — `planBatch`/`runTier`/`aggregateMarkers`/`commitCommand`, reused not rewritten), `specs/03-profile-commands-spec.md` (the HV verify flow); `specs-reference/infrastructure/I00` index + `I01`–`I09` (the per-artifact production specs — the headless session's system-prompt body, loaded via `COVERAGE_MAP[id].specPath` exactly like docs 01–12). `@upriver/schemas`' `COVERAGE_MAP` already carries `i01`–`i09` with `requiresFields`/`requiresHvVerified`/`requiresDocs` and DAG edges (I07 first; I01 needs all 18 docs + I07; then I02→I04→I03; I05/I06/I08/I09 parallel after I01) — read-only for this build. ## 1. Purpose and scope

`upriver generate <slug> --all --provisioning [--docs <i-ids>] [--from <i-id>] [--dry-run] [--yes]` runs the **i-series** (I01–I09) through the exact M2 batch loop, and `upriver generate <slug> --doc <i-id>` produces a single provisioning artifact through the single-doc engine. Both reuse the proven pipeline: load + gate once (readiness = required fields present ∧ required HV fields verified ∧ upstream deliverables *approved*), DAG-tier by `generationOrder`, generate one write-capable headless session per artifact against the artifact's production spec (`COVERAGE_MAP[id].specPath` → `specs-reference/infrastructure/I0X-*.md`), scan markers, gate per tier, print the per-gate commit. The deltas are entirely additive and small:

1. **Scope gate widened to the i-series.** The engine's hard scope gate (`M1_DOCS.includes(id)`, the line that today rejects provisioning ids with "land in later specs") widens to `M1_DOCS ∪ I_SERIES`. `I_SERIES = [i01 … i09]`; docs 13–18 stay out of scope (operational, generated rarely — out of the per-client provisioning map). `--all --provisioning` feeds `planBatch` the i-series scope instead of `ALL_DOCS`; everything downstream (tiering, per-tier gate, commit) is unchanged.

2. **The "docs approved" precondition is surfaced, not bypassed.** Each i-artifact's `requiresDocs` includes prose docs (I01 needs all 18; I02 needs doc-01/03/04/06/07/11; etc.) plus i-series edges (I01 needs I07; I02 needs I01; I04 needs I02; I03 needs I02+I04). When the scope is the i-series, those doc deps are *out-of-scope* and must already be **approved** in the manifest, or the artifact is reported `blocked-upstream` (kind `unapproved-out-of-scope`) — exactly the existing `planBatch` classification. The operator's path is therefore: `generate <slug> --all` (M2, approve the docs) **then** `generate <slug> --all --provisioning`. The command prints a hint to that effect when provisioning blocks on unapproved docs.

3. **Output shape: one artifact, three parts, two marker classes (PRD §6).** Each I0X spec declares a single named reference markdown (`[client]-…​.md`, its "File naming" row) whose body is a section template **plus** a "How to Build This Document" operator runbook **plus** a Definition-of-Done checklist of click-ops. The headless session (one session → one file, the existing `runDoc` model) writes that single artifact containing: (A) the generated configuration/reference per the spec's section template; (B) an **operator runbook** of the in-account steps in order; (C) an **operator-must-do** checklist for what cannot be generated. The model emits **two** inline marker classes:
   - `[NEEDS CONFIRMATION: <question>]` — thin/ambiguous profile (the existing class, unchanged).
   - `[OPERATOR ACTION: <step>]` — a step that requires the client's Anthropic account and so **cannot be generated** from a file: Project creation, OAuth consent screens, skill ZIP upload, plan-tier/billing change, extension install, artifact publishing. This is the second marker class; the engine scans it, never attempts it, and the run report aggregates it into the operator-must-do checklist.

4. **HV is densest here, and the gate is the only path through it (PRD §6).** OAuth scopes, allowlists/banned sites (I05), API spend caps (I06), plan tier + billing (I07), data residency/compliance (I04/I07), publishing approvals (I09) are all `humanVerifyRequired`. No artifact is generated while any of its required HV fields is unverified — `deliverableReadiness` blocks it and `planBatch` leaves it `blocked` with the named unverified-HV fields. The only path through is `upriver profile verify` (M4). `verified:true` is operator-only; no artifact consumes an unverified HV value (generation is gated on `readiness.ready`, which requires `unverifiedHv === []`).

I07 is the DAG-first, near-fully-HV decision document (its spec: "build first of the client-facing specs"; governance gates everything). In `littlefriends` most i-series fields are unverified HV or missing, so the gate demonstrably blocks the whole i-series until the operator fills + verifies — the doc-02 analogue, one tier up.

**Resolved decisions** (full rationale in the BUILT changelog):
- **Surface = extend the M2 batch** (not a sibling `--provision` command). `--all --provisioning` reuses `runAll`'s tier loop, per-tier gate, `--from`/`--yes`/`--dry-run`, and the printed commit verbatim; the only change is the scope set fed to `planBatch` and a scope-aware `--docs` validation. Single-artifact runs use `--doc <i-id>` (the engine gate widens, so it works for free) or `--all --provisioning --docs <i-id>`.
- **Output shape = one combined markdown per artifact**, three parts (config + runbook + cannot-generate checklist), two inline marker classes — confirmed against each I0X spec's single-file "File naming"/"Delivery format" declaration rather than invented. The operator-action marker is the second class alongside `[NEEDS CONFIRMATION]`, aggregated into the run report.

**Out of scope:** I10–I15 (Joshua's own internal infrastructure — out of the per-client map, PRD §3.4); generation of operational docs 13–18 (an I01 prerequisite the engine still does not produce — a noted follow-up, not this build); the worker enqueue allowlists + worktree parallelism (M8); any change to how a single deliverable is generated, gated, or committed.
