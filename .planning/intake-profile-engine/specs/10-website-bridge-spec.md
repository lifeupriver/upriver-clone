# Build Spec 10: the profile→website bridge (gap G6)

Component: connect the Client Profile spine to the existing website pipeline so the "modern website" half of the end-state runs off the same profile as the AI Operating System half. Three pieces: (A) a `doc-web-prd` deliverable (the website requirements doc, generated like any other doc), (B) profile-first inputs for the existing `design-brief` → `scaffold` → `clone` → `improve` pipeline with audit-package fallback, and (C) a `design-system` deliverable for the new-design path. Closes capability-audit gap G6; touches G1's adjacency (the web-PRD feeds proposals later).
Source of truth: `08-capability-audit.md` G6; `01-prd.md` §7 (one profile, one spine); the end-state capability map stages 8–9. The existing pipeline's behavior is the compatibility contract: a client with an audit-package and no profile must keep working byte-identically.

## Two paths, one bridge (the decision structure this spec encodes)

The end-state names two website outcomes, both downstream of the deep audit:

- **Clone path** (site is structurally fine, platform is the problem — e.g. Square → Astro): `scaffold` + `clone` rebuild the existing site page-by-page. EXISTS; this spec repoints its content inputs at the profile.
- **New-design path** (site is a brand liability): design system → new build from the web-PRD. The design-system *document* is generatable; the new-site *build* remains operator-driven Claude Code work against the scaffold-template, guided by the web-PRD + design system (generating a whole bespoke site unattended is out of scope and stays out — the Continue-gate philosophy applies to whole sites doubly).

Which path applies is the doc-10 §9 scope fork (`goals.engagementScope.websiteScope`: A iterative / B partial rebuild / C full rebuild) — already an HV-adjacent profile field. The bridge reads it; it never decides it.

## B. Profile-first inputs for the existing pipeline

One seam, applied identically in three commands. Build `web-bridge/inputs.ts`:

```ts

export async function resolveWebInputs(slug: string): Promise<WebInputs>
// WebInputs: { source: 'profile' | 'audit-package' | 'merged',
//   voice, facts, offerings, seoTargets, brandTokens, pages, intakeDecisions }
```

Rules: if `profile.json` exists and `clientProfileZ` parses, fields come from the profile (voice section → brand voice; identity+offerings+pricing-visibility → facts; seo section → targets; content.visualBrand → tokens) and `auditDecisions` replaces the legacy intake read; `siteStructure.pages` + screenshots still come from the audit-package (they are scrape artifacts, not profile data) — hence `'merged'` is the normal source for a full engagement. No profile → pure audit-package path, byte-identical to today (regression-tested). `design-brief.ts`, `scaffold/index.ts`, `clone.ts` each swap their direct `loadAuditPackage`/`readIntake` content reads for `resolveWebInputs` at the defined call sites only; their generation/rendering logic is untouched. Generated pages MUST source copy facts from the profile when present — the point of the bridge is that the clone never re-invents facts the spine already verified (capacity numbers, hours, pricing visibility policy all flow from HV-verified fields).

## C. `design-system` — the new-design path's generatable half

1. Author `specs-reference/ai-operating-system/20-design-system-spec.md`: inputs (voice attributes + tone spectrums from doc-01/profile, visual brand assets from content section, design pass findings from the audit, competitor visual differentiation from doc-05); output = a design-system document (type scale, palette w/ hex from existing brand or proposed, spacing, component inventory, photography direction, do/don't examples) PLUS machine-readable tokens emitted as `clients/<slug>/design-system/tokens.json` (the scaffold-template's global.css consumes a compatible shape — verify and document the mapping, don't refactor the template). External tools (Claude Design sessions, Pinterest refs, Gemini logo concepts) are OPERATOR steps: the doc's runbook section lists them as `[OPERATOR ACTION: …]` exactly like provisioning runbooks; the generator never calls external design tools.
2. `COVERAGE_MAP` row `design-system`: requires voice, content.visualBrand, positioning; `requiresDocs` doc-01, doc-05, doc-10; scoped to `--web`; gated on the same `websiteScope` HV field (only B/C scopes need it; the spec's system prompt receives the scope value and says so).

  **Deviations from the spec letter (all to honor its constraints):**
  - `WebInputs` carries `pkg` and `intakeDecisions` in addition to the spec's listed resolved fields (voice/facts/offerings/seoTargets/brandTokens/pages). All three commands consume the full `AuditPackage` shape extensively, so a carrier `pkg` is the only way to keep the seam a one-line swap with NO other diff and byte-identical output.
